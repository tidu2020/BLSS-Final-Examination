# 01 系统需求分析

---

## 1.1 项目背景

国有企业在法务合规管理中日常面临两类痛点：

- **业务侧**：日常法务咨询（如"合同相对方要求第三方付款可以吗""证据逾期提交后果"）缺少前置自助渠道，小事也需打扰法务；正式材料（合同/对外函件/采购/上会议题）审核流转缺乏统一入口。
- **法务侧**：审核经验以文档形式散落，难以沉淀复用；年度法治工作总结、合规季报、风险报告等公文重复起草；缺乏对历史审核数据的结构化利用，难以做风险监测与年度风险识别。

本系统构建"业务-法务协同 + AI 辅助审核 + 知识自动沉淀"的前后端分离 Web 闭环平台，支撑国有企业法务合规"三个100%"（重大决策/规章制度/合同法律审核率 100%）与"四位一体"（法务/合规/内控/风控融合）体系落地。

| 轨道 | 约束 | 设计应对 |

## 1.3 功能需求

系统采用前后端分离架构，**每个功能模块均包含独立的后端 API 和前端页面**。业务、法务、管理员三套前端界面各不相同。

### 核心模块

| 编号 | 模块 | 后端 API | 前端 | 说明 |
|------|------|---------|------|------|
| ① | 账户与权限 | `/api/auth/*` | 登录页 | 三角色登录、鉴权 |
| ② | 业务门户 | `/api/business/*` | 业务端 | 发起会话、上传材料、查看 AI 意见、提交工单 |
| ③ | 法务工作台 | `/api/legal/*` | 法务端 | 接收工单、审核、归档前审核修改、校验、确认入库 |
| ④ | AI 审核引擎 | `/api/ai/review` | （被业务门户调用） | 双算法检索 + LLM 生成、多轮对话、免责声明 |
| ⑤ | 知识库管理 | `/api/kb/*` | 管理员端 | 管理员直接管理知识库；法务仅检索调用 |
| ⑥ | 公文生成 | `/api/report/*` | 法务/管理员 | 基础版：1-2 类公文 |
| ⑦ | 归档导出 | `/api/archive/*` | 管理员 | Markdown 导出 + 按案例归档 |

### 系统更新计划（不在 MVP 实现）

| 模块 | 说明 |
|------|------|
| 风险监测引擎 | 扫描知识库识别近期高发问题 |
| 年度风险报告 | 基于风险监测数据生成 |
| 知识出书 | 按主题聚合案例集出版 |
| 审核留痕审计日志 | AuditLog 完整状态变更链 |

## 1.4 权限模型（三角色）

| 角色 | 权限 | 前端界面 |
|------|------|---------|
| **业务**（business） | 发起咨询、上传材料、查看 AI 意见、提交法务工单、查看自己的工单状态 | 业务端 |
| **法务**（legal） | 接收工单、审核、**调用知识库检索**（不可直接增删改）、归档前审核修改、校验、确认入库 | 法务端 |
| **管理员**（admin） | **直接查看/编辑/删除知识库条目**、管理账户、浏览归档、查看全部工单数据 | 管理员端 |

**权限边界说明**：
- 法务**不能直接修改知识库**，只能在"归档前审核"环节对拟入库条目做修改，经**校验**后确认入库
- 管理员可直接操作知识库（增删改查），拥有最高权限
- 业务只能看到自己提交的工单，法务和管理员可看全部

## 1.5 校验功能（法务归档前）

法务在确认入库前，系统自动对拟入库条目执行 6 项校验：

| 校验项 | 规则 | 类型 | 失败处理 |
|--------|------|------|---------|
| 问题完整性 | question 字段非空（**有问题才能有答案**） | 硬 | 阻止入库，提示补全问题 |
| 四段式完整性 | 四个字段（法律解答/合规风险/实操建议/相关法条）均非空 | 硬 | 阻止入库，提示补全 |
| 法条格式 | `legal_basis` 含《》及"第X条"格式 | 软 | 警告，可强制入库 |
| 标签完整性 | 至少 1 个 tag | 软 | 警告，可强制入库 |
| 重复校验 | 与现有知识库条目相似度 > 0.8 | 软 | 警告，显示相似条目供法务判断 |
| 问题格式 | question 以"？"结尾 | 软 | 警告，可强制入库 |

- **硬错误**（errors）：阻止入库
- **软警告**（warnings）：可强制入库（需法务二次确认）

## 1.6 AI 审核引擎需求

| 需求项 | 说明 |
|--------|------|
| 检索方式 | 双算法：KeywordMatcher 粗筛 Top-20 + TF-IDF 余弦相似度精排 Top-5 |
| 生成方式 | RAG + LLM：将检索结果作为上下文，调用大模型生成四段式法务意见 |
| 回退机制 | LLM 不可用/失败时，自动回退到规则拼装（纯 RAG） |
| 大模型 | OpenAI 兼容协议；测试用豆包 Ark `ark-code-latest`，生产换私有部署 |
| 免责声明 | 每条 AI 回复自动附加免责声明，提示需提交法务获取正式意见 |
| 多轮对话 | 业务可在同一会话内连续咨询，上下文保留 |

## 1.8 非功能需求

- **数据安全**：MVP 阶段知识库本地存储（JSON）；企业部署时合同/函件等涉密材料禁止出公司网络，大模型走私有部署。
- **可演示性**：浏览器访问 `http://localhost:5000`，单条 FAQ 检索响应 < 1 秒（LLM 生成约 3-8 秒）。
- **可维护性**：前后端分离，后端模块化路由，前端按角色独立页面。
- **可扩展性**：知识库存储层抽象，未来可替换为 SQLite/向量库；前端可替换为 React/Vue；大模型可通过环境变量切换。
- **无外部依赖**：后端仅需 Flask，算法用标准库自研实现（符合课程要求）。
