   - 自动生成 id、month、category、tags

5. **权限装饰器**（`auth/decorator.py`）
   - `@require_role("business")` 等三角色鉴权

6. **校验器**（`legal/validator.py`）
   - 6 项校验全部实现，硬/软分级

7. **工单留痕**（`models/workflow.py`）
   - WorkOrder 含提交人/时间、AI 结论、审核人/时间、入库来源全链路

8. **归档导出**（`archive/`）
   - 法务确认后打包 Markdown 归档

**遇到的问题与解决**：

### 阶段四：大模型集成（2026-06-27）

**目标**：先用用户自有大模型测试，生产环境换公司私有部署。

**用户提供的配置**：
- 模型：`ark-code-latest`
- Base URL：`https://ark.cn-beijing.volces.com/api/coding/v3`（OpenAI 兼容协议，支持 Responses API）
- API key（初版）：`apikey-20260627001314-cl7qq`

**完成事项**：

1. **大模型客户端**（`ai/llm_client.py`）
   - 用 Python 标准库 `urllib` 实现 OpenAI 兼容客户端
   - 不引入 openai SDK，保持零外部依赖
   - `LlmClient.chat(messages)` 调用 `/chat/completions` 端点

2. **配置管理**（`config.py`）
   - 新增 LLM 配置项，环境变量优先：
     ```python
     LLM_API_KEY = os.environ.get("LLM_API_KEY", "默认测试key")
     LLM_BASE_URL = os.environ.get("LLM_BASE_URL", "https://ark.cn-beijing.volces.com/api/coding/v3")
     LLM_MODEL = os.environ.get("LLM_MODEL", "ark-code-latest")
     LLM_TEMPERATURE = 0.3  # 法务场景取低值，保证严谨
     LLM_MAX_TOKENS = 1024
     ```

3. **RAG+LLM 编排器改造**（`ai/orchestrator.py`）
   - 通过依赖注入接收 `llm_client`
   - `_generate()` 方法：LLM 优先，失败自动回退规则拼装
   - `_generate_with_llm()`：将 Top-K 检索结果作为上下文，约束模型只依据知识库作答
   - System Prompt 约束：分四段、不编造、指出需进一步咨询法务的边界

4. **应用注入**（`app.py`）
   - `create_app()` 中初始化 LlmClient 并注入 ReviewOrchestrator
   - 启动时打印 LLM 状态（已启用/未配置）

**遇到的问题与解决**：

| 问题 | 原因 | 解决 |
|------|------|------|
| HTTP 401 API key 格式错误 | `apikey-20260627001314-cl7qq` 不符合 Ark key 格式 | 用户补充正确 key `ark-4271b198-...`（UUID 格式）；代码层增加回退机制保证健壮性 |
| 代码改了不生效 | `__pycache__` 缓存旧字节码 | 重启前清理 `__pycache__` |

**验证结果**：
- LLM 连通性测试：✅ 成功，返回专业的第三方付款法律意见
- RAG+LLM 端到端测试：✅ 成功
  - candidates: 20
  - Top1 命中：合同第三方付款需要注意什么？
  - 回复 463 字，四段式结构完整
  - 引用法条：《民法典》第五百一十四条
  - 自动附加免责声明
- 回退机制测试：✅ 4 类异常（401/超时/空响应/未配置）均优雅降级

**关键设计决策**：
- **依赖注入**：LLM 客户端通过构造函数注入 ReviewOrchestrator，便于测试和生产替换
- **优雅降级**：LLM 不可用时不阻断服务，回退纯 RAG 模式，保证系统可用性
- **知识库约束**：System Prompt 明确要求"严格依据知识库案例回答，不得编造"，降低幻觉风险
- **环境变量优先**：生产环境用环境变量注入密钥，不写进源码

## 关键技术决策汇总

| # | 决策点 | 选择 | 理由 |
|---|--------|------|------|
| 1 | 算法实现 | 自研 TF-IDF + 余弦 | 课程要求体现算法原理理解 |
| 2 | 存储方案 | JSON 文件 | 无依赖、易演示、可抽象替换 |
| 3 | 前端技术 | 原生 HTML/JS | 无构建工具，符合实验要求 |
| 4 | LLM 客户端 | urllib 标准库实现 | 零外部依赖，兼容 OpenAI 协议 |
| 5 | LLM 集成方式 | 依赖注入 + 优雅降级 | 模块化、可测试、生产可替换 |
| 6 | 权限模型 | 三角色 + 装饰器 | 边界清晰，法务不可直接改知识库 |
| 7 | 校验分级 | 硬错误阻止 / 软警告可强制 | 兼顾严谨性与效率 |
| 8 | 配置管理 | 环境变量优先 | 生产环境密钥不进源码 |
| 9 | RAG 上下文约束 | System Prompt 限制 | 降低 LLM 幻觉，确保依据知识库 |
| 10 | 回复结构 | 四段式 + 免责声明 | 对应业务需求，提示需法务正式意见 |
