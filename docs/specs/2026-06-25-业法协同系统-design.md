## 一、实验内容与系统需求分析

### 1.1 项目背景

国有企业在法务合规管理中日常面临两类痛点：

- **业务侧**：日常法务咨询（如"合同相对方要求第三方付款可以吗""证据逾期提交后果"）缺少前置自助渠道，小事也需打扰法务；正式材料（合同/对外函件/采购/上会议题）审核流转缺乏统一入口。
- **法务侧**：审核经验以文档形式散落，难以沉淀复用；年度法治工作总结、合规季报、风险报告等公文重复起草；缺乏对历史审核数据的结构化利用，难以做风险监测与年度风险识别。

本系统旨在构建一个"业务-法务协同 + AI 辅助审核 + 知识自动沉淀"的前后端分离 Web 闭环平台，支撑国有企业法务合规"三个100%"（重大决策/规章制度/合同法律审核率 100%）与"四位一体"（法务/合规/内控/风控融合）体系落地。

### 1.2 双轨定位约束

| 轨道 | 约束 | 设计应对 |
|------|------|---------|
| 课程实验 | Python 代码统一 .py 格式；OOP；自研算法；完整文档 | 后端全 Python（Flask + 标准库）；TF-IDF 自实现；类层次清晰 |
| 企业部署 | 账户权限分级；涉密数据不上云；前后端分离可扩展 | 三角色权限模型；知识库本地化；REST API + 独立前端 |

### 1.3 功能需求（MVP 范围）

系统采用前后端分离架构，**每个功能模块均包含独立的后端 API 和前端页面**。业务、法务、管理员三套前端界面各不相同。

**核心模块（必须实现）：**

| 编号 | 模块 | 后端 | 前端 | 说明 |
|------|------|------|------|------|
| ① | 账户与权限模块 | `/api/auth/*` | 登录页 | 三角色（业务/法务/管理员）登录、鉴权 |
| ② | 业务门户模块 | `/api/business/*` | 业务端界面 | 发起会话、上传材料、查看 AI 意见、提交工单 |
| ③ | 法务工作台模块 | `/api/legal/*` | 法务端界面 | 接收工单、审核、归档前审核修改、校验、确认入库 |
| ④ | AI 审核引擎 | `/api/ai/review` | （被业务门户调用，无独立页面） | 双算法检索、多轮对话、免责声明 |
| ⑤ | 知识库管理模块 | `/api/kb/*` | 管理员端界面 | 管理员直接管理知识库；法务仅检索调用 |
| ⑥ | 公文生成模块 | `/api/report/*` | 法务/管理员界面 | 基础版：1-2 类公文 |
| ⑦ | 归档导出模块 | `/api/archive/*` | 管理员界面 | Markdown 导出 + 按月份/案例归档 |

**写入"系统更新计划"（不在 MVP 实现）：**

| 模块 | 说明 |
|------|------|

| 风险监测引擎 | 扫描知识库识别近期高发问题 |
| 年度风险报告 | 基于风险监测数据生成 |

### 1.4 权限模型（三角色）

| 角色 | 权限 | 前端界面 |
|------|------|---------|
| **业务**（business） | 发起咨询、上传材料、查看 AI 意见、提交法务工单、查看自己提交的工单状态 | 业务端 |
| **法务**（legal） | 接收工单、审核、**调用知识库检索**（不可直接增删改）、归档前审核修改、校验、确认入库 | 法务端 |
| **管理员**（admin） | **直接查看/编辑/删除知识库条目**、管理账户、浏览归档、查看全部工单数据 | 管理员端 |

**权限边界说明**：
- 法务**不能直接修改知识库**，只能在"归档前审核"环节对拟入库条目做修改，经**校验**后确认入库
- 管理员可直接操作知识库（增删改查），拥有最高权限
- 业务只能看到自己提交的工单，法务和管理员可看全部

### 1.5 校验功能（法务归档前）

法务在确认入库前，系统自动对拟入库条目执行校验：

| 校验项 | 规则 | 失败处理 |
|--------|------|---------|
| **问题完整性** | question 字段非空（**有问题才能有答案**） | 阻止入库，提示补全问题 |
| 四段式完整性 | 四个字段（法律解答/合规风险/实操建议/相关法条）均非空 | 阻止入库，提示补全 |
| 法条格式 | `legal_basis` 含《》及"第X条"格式 | 警告，可强制入库 |
| 标签完整性 | 至少 1 个 tag | 警告，可强制入库 |
| 重复校验 | 与现有知识库条目相似度 > 0.8 | 警告，显示相似条目供法务判断 |
| 问题格式 | question 以"？"结尾 | 警告，可强制入库 |

### 1.6 非功能需求

- **数据安全**：MVP 阶段全部本地存储（JSON），不调用云端大模型 API；企业部署时合同/函件等涉密材料禁止出公司网络。
- **可演示性**：浏览器访问 `http://localhost:5000`，单条 FAQ 检索响应 < 1 秒。
- **可维护性**：前后端分离，后端模块化路由，前端按角色独立页面。
- **可扩展性**：知识库存储层抽象，未来可替换为 SQLite/向量库；前端可替换为 React/Vue。

### 1.7 合规与留痕要求（设计预留，MVP 不实现完整功能）

企业法务场景要求审核过程全程可追溯，本系统在设计上为以下合规要求预留数据基础：

| 留痕要素 | 设计预留位置 | MVP 状态 |
|---------|------------|---------|
| 提交人/提交时间 | WorkOrder.submitter / submitted_at | ✅ 已设计 |
| AI 审核意见及生成时间 | WorkOrder.ai_conclusion / ai_reviewed_at | ✅ 已设计 |
| 法务审阅人/审阅时间 | WorkOrder.reviewer / reviewed_at | ✅ 已设计 |
| 法务最终结论 | WorkOrder.legal_conclusion | ✅ 已设计 |
| 入库条目来源工单 | FAQ.source_work_order_id | ✅ 已设计 |
| 状态变更全记录 | AuditLog（完整留痕日志） | 📝 下一步计划 |

### 4.1 账户与权限模块

**后端（auth/）**：

```python
# models/account.py
class Account(ABC):
    def __init__(self, user_id, name, pwd_hash):
        self.id = user_id
        self.name = name
        self._pwd_hash = pwd_hash

    @abstractmethod
    def role(self) -> str: ...

    def login(self, pwd: str) -> bool:
        return self._pwd_hash == hashlib.sha256(pwd.encode()).hexdigest()

    def can_review(self) -> bool: return False
    def can_manage_kb(self) -> bool: return False
    def can_view_all_orders(self) -> bool: return False

class BusinessUser(Account):
    def role(self): return "business"
    def can_submit(self): return True

class LegalUser(Account):
    def role(self): return "legal"
    def can_review(self): return True
    def can_confirm(self): return True
    def can_search_kb(self): return True   # 可检索，不可增删

class AdminUser(Account):
    def role(self): return "admin"
    def can_review(self): return True
    def can_manage_kb(self): return True  # 可直接增删改
    def can_view_all_orders(self): return True
```

```python
# auth/decorator.py
def require_role(*roles):
    """路由权限装饰器。"""
    def decorator(f):
        @functools.wraps(f)
        def wrapper(*args, **kwargs):
            user = session.get("user")
            if not user or user["role"] not in roles:
                return jsonify({"error": "权限不足"}), 403
            return f(*args, **kwargs)
        return wrapper
    return decorator
```

```python
# auth/routes.py
@bp.route("/login", methods=["POST"])
def login():
    user_id = request.json["user_id"]
    pwd = request.json["password"]
    user = account_store.get(user_id)
    if user and user.login(pwd):
        session["user"] = {"id": user.id, "role": user.role()}
        return jsonify({"role": user.role()})
    return jsonify({"error": "账号或密码错误"}), 401
```

### 4.3 法务工作台模块（含校验）

**后端（legal/）**：

```python
@bp.route("/orders")
@require_role("legal", "admin")
def list_orders():
    """待审核工单列表。"""
    orders = work_order_store.query(status="submitted_to_legal")

@bp.route("/review/<order_id>", methods=["GET", "POST"])
@require_role("legal", "admin")
def review(order_id):
    order = work_order_store.get(order_id)
    if request.method == "POST":
        order.legal_conclusion = request.json["conclusion"]
        order.reviewer = session["user"]["id"]
        order.reviewed_at = now()
        return jsonify({"status": "saved"})
    return jsonify(order.to_dict())

@bp.route("/validate", methods=["POST"])
@require_role("legal", "admin")
def validate_entry():
    """归档前校验拟入库条目。"""
    item = request.json
    validator = FaqValidator(knowledge_base)
    result = validator.validate(item)
    return jsonify(result)  # {passed: bool, warnings: [], errors: []}

@bp.route("/confirm", methods=["POST"])
@require_role("legal", "admin")
def confirm_to_kb():
    """法务确认入库（需先通过校验）。"""
    item = request.json
    validator = FaqValidator(knowledge_base)
    result = validator.validate(item)
    if result["errors"]:  # 有硬错误则阻止
        return jsonify({"error": "校验未通过", "detail": result["errors"]}), 400
    item["source_work_order_id"] = request.json["order_id"]
    knowledge_base.add(item)
    return jsonify({"status": "confirmed", "faq_id": item["id"]})
```

```python
class FaqValidator:
    """法务归档前校验器。"""

    def validate(self, item: dict) -> dict:
        errors = []    # 硬错误，阻止入库
        warnings = []  # 软警告，可强制

        # 1. 问题完整性（硬）——有问题才有答案
        if not item.get("question", "").strip():
            errors.append("问题 question 为空，必须先有问题才能作答")

        # 2. 四段式完整性（硬）
        for f in ["legal_answer", "compliance_risk",
                   "practical_advice", "legal_basis"]:
            if not item.get(f, "").strip():
                errors.append(f"字段 {f} 为空")

        # 3. 法条格式（软）
        if item.get("legal_basis"):
            if not re.search(r"《.+?》.*?第.+?条", item["legal_basis"]):
                warnings.append("法条格式不规范，建议含《法名》第X条")

        # 4. 标签完整性（软）
        if not item.get("tags"):
            warnings.append("未设置标签，建议至少 1 个")

        # 5. 重复校验（软）
        if item.get("question"):
            existing = knowledge_base.search(item["question"], top_k=1)
            if existing and SimilarityRetriever._cosine(
                    self._vec(item["question"]),
                    self._vec(existing[0]["question"])) > 0.8:
                warnings.append(f"与已有条目 {existing[0]['id']} 高度相似")

        # 6. 问题格式（软）
        if not item.get("question", "").rstrip().endswith(("？", "?")):
            warnings.append("问题未以问号结尾")

        return {
            "passed": len(errors) == 0,
            "errors": errors,
            "warnings": warnings
        }
```

## 五、核心算法设计

### 5.1 算法一：KeywordMatcher（关键词粗筛）

**目标**：从 67 条 FAQ 中快速筛出 Top-20 候选。

**原理**：扩展关键词集合求交集思路，增加法务术语词典。

```python
class KeywordMatcher:
    STOP_WORDS = {"什么", "是", "的", "在", "和", "与", "及", "了",
                  "吗", "呢", "啊", "作用", "区别", "之间", "中",
                  "等", "主要", "通常", "一般", "如何", "怎么"}

    LEGAL_TERMS = {"合同", "担保", "诉讼", "仲裁", "合规", "内控",
                   "风控", "证据", "管辖", "侵权", "著作权", "商标",
                   "专利", "租赁", "保证金", "代理", "国企", "发票"}

    def extract(self, text):
        tokens = re.findall(r"[A-Za-z0-9]+|[\u4e00-\u9fa5]", text)
        cn = [t for t in tokens if re.match(r"[\u4e00-\u9fa5]", t)]
        kws = set()
        for ch in cn:
            if ch not in self.STOP_WORDS: kws.add(ch)
        for i in range(len(cn)-1):
            bi = cn[i] + cn[i+1]
            if bi not in self.STOP_WORDS: kws.add(bi)
        for term in self.LEGAL_TERMS:
            if term in text: kws.add(term)
        return kws

    def filter(self, query, items, top_n=20):
        q_kws = self.extract(query)
        scored = []
        for item in items:
            item_kws = self.extract(item["question"] + item["legal_answer"])
            score = len(q_kws & item_kws)
            if score > 0:
                scored.append((score, item))
        scored.sort(key=lambda x: -x[0])
        return [it for _, it in scored[:top_n]]
```

### 5.2 算法二：SimilarityRetriever（TF-IDF 精排）

**目标**：对粗筛候选集用 TF-IDF + 余弦相似度精排，返回 Top-5。

**算法原理**：

1. **TF（词频）**：词 t 在文档 d 中出现次数 / 文档总词数
2. **IDF（逆文档频率）**：`log(N / (df + 1))`，N 为文档总数，df 为含 t 的文档数
3. **TF-IDF 向量**：每篇文档表示为各词 TF-IDF 值的向量
4. **余弦相似度**：`cos(θ) = (A·B) / (|A|·|B|)`

**自行实现**（不依赖 sklearn）：

```python
class SimilarityRetriever:
    def __init__(self):
        self.vocab: list[str] = []
        self.idf: dict[str, float] = {}
        self.doc_vectors: list[list[float]] = []

    def fit(self, items):
        """构建词汇表和 IDF。"""
        docs = [it["question"] + " " + it["legal_answer"] for it in items]
        N = len(docs)
        df = collections.Counter()
        for doc in docs:
            for w in set(self._tokenize(doc)):
                df[w] += 1
        self.vocab = list(df.keys())
        self.idf = {w: math.log(N / (df[w] + 1)) for w in self.vocab}
        self.doc_vectors = [self._vectorize(d) for d in docs]
        self.items = items

    def _vectorize(self, doc):
        tf = collections.Counter(self._tokenize(doc))
        return [tf.get(w, 0) * self.idf.get(w, 0) for w in self.vocab]

    def rank(self, query, candidates, top_k=5):
        q_vec = self._vectorize(query)
        scored = []
        for cand in candidates:
            idx = self.items.index(cand)
            score = self._cosine(q_vec, self.doc_vectors[idx])
            scored.append((score, cand))
        scored.sort(key=lambda x: -x[0])
        return [it for _, it in scored[:top_k]]

    @staticmethod
    def _cosine(a, b):
        dot = sum(x*y for x, y in zip(a, b))
        na = math.sqrt(sum(x*x for x in a))
        nb = math.sqrt(sum(y*y for y in b))
        return dot / (na * nb + 1e-10)

    def _tokenize(self, text):
        return re.findall(r"[A-Za-z0-9]+|[\u4e00-\u9fa5]", text)
```

**为什么自研而非调 sklearn**：
- 实验要求"算法逻辑需自行实现或合理封装，体现对算法原理的理解"
- 自研可清晰展示 TF/IDF/余弦公式的代码实现
- 自研版作为基线，后续可与 sklearn 版做性能对比（加分项）

### 5.3 编排器（ReviewOrchestrator）

```python
class ReviewOrchestrator:
    DISCLAIMER = (
        "\n\n⚠️ 本意见由 AI 基于知识库生成，仅供参考。"
        "未提交法务处理，不得作为正式决策依据。"
        "如需正式法律意见，请提交法务处理。"
    )

    def __init__(self, kb: KnowledgeBase):
        self.kb = kb
        self.matcher = KeywordMatcher()
        self.retriever = SimilarityRetriever()
        self.retriever.fit(kb.items)

    def review(self, user_input: str) -> str:
        candidates = self.matcher.filter(user_input, self.kb.items)
        if not candidates:
            return "未检索到相关知识。" + self.DISCLAIMER
        ranked = self.retriever.rank(user_input, candidates)
        return self._compose(ranked) + self.DISCLAIMER

    def _compose(self, items) -> str:
        parts = []
        for i, it in enumerate(items, 1):
            parts.append(
                f"【参考{i}】{it['question']}\n"
                f"  法律解答：{it['legal_answer'][:200]}\n"
                f"  合规风险：{it['compliance_risk'][:150]}\n"
                f"  实操建议：{it['practical_advice'][:200]}\n"
                f"  依据法条：{it['legal_basis'][:150]}"
            )
        return "\n\n".join(parts)
```

```

   │
   ↓

   │
   ↓

   │
   ↓

   │
   ↓

   │
   ↓

   │
   ↓
按相似度排序取 Top-5
   │
   ↓
[ReviewOrchestrator] 拼装四段式回复 + 免责声明
   │
   ↓
返回前端展示
```

**当前状态**：WorkOrder 已记录关键节点的操作人和时间戳，但未记录完整状态变更链。

**下一步目标**：实现完整审计日志，满足企业法务合规要求。

**实现方案**：
- 新增 AuditLog 类，记录工单每一次状态变更：
  ```json
  {
    "work_order_id": "WO-2026-0001",
    "event": "status_change",
    "from_state": "ai_reviewed",
    "to_state": "submitted_to_legal",
    "operator": "business01",
    "timestamp": "2026-06-25T10:30:00",
    "detail": "业务提交法务工单"
  }
  ```
- WorkOrder 的所有状态变更方法自动写入 AuditLog
- 支持按工单 ID 查询完整操作轨迹
- 支持按操作人/时间范围查询（满足审计检查场景）

**触发场景**：
- 内部合规检查：调取某工单完整操作链
- 责任追溯：某入库条目争议时，回溯来源工单的审核全过程
- 监管报送：年度法治工作总结时，统计审核流转数据

### 9.8 电子档案归档（ArchivePackager）

**当前状态**：工单数据以 JSON 散存于 work_orders 目录，无统一归档。

**下一步目标**：法务确认入库后，系统自动整理电子档案归档。

**归档触发流程**：
```
法务确认入库
     │
     ↓
[ArchivePackager] 自动打包
     │
     ├─ 生成归档编号（如 ARCH-2026-06-0001）
     ├─ 整理工单全量数据（材料+对话+AI结论+法务结论+入库FAQ）
     ├─ 按分类组织（合同/函件/采购/议题/咨询）
     ├─ 生成归档目录索引（archive_index.json）
     └─ 写入 data/archive/{归档编号}/
     │
     ↓
归档完成，WorkOrder.status = "archived"
```

**归档内容**：
```
data/archive/ARCH-2026-06-0001/
├─ manifest.json          # 归档清单（编号/分类/时间/相关人员）
├─ materials/             # 原始材料副本
├─ dialogue.json          # 对话记录
├─ ai_conclusion.txt      # AI 审核意见
├─ legal_conclusion.txt   # 法务最终结论
├─ confirmed_faqs.json    # 入库的知识条目
└─ audit_trail.json       # 完整操作日志（依赖 9.7）
```

**实现方案**：
- 新增 ArchivePackager 类（继承 ResultExporter）
- 在法务确认入库后自动触发归档
- 归档支持按月/季/年/部门/类型多维度检索
- 支持导出归档清单为 Excel（满足档案管理要求）

**价值**：
- 满足企业电子档案管理合规要求
- 年度法治工作总结时可直接调取归档数据
- 支撑"制度流程化、流程信息化"理念落地

## 十、遇到的问题及解决方案

### 10.1 FAQ 数据格式不规范

**问题**：原始 FAQ.docx 是 Word 二进制，内容为"问题→单段答案"，无三段式结构，无法直接支撑分层输出。

**解决方案**：
1. 第一轮：人工重写为三段式（法律解答/合规风险/实操建议），存为 FAQ_重写版.md
2. 第二轮：发现缺法条依据，补充第四段"相关法条"，存为 FAQ_四部分版_最终.md
3. 编写 DataPreprocessor 类自动解析 Markdown 结构化为 JSON

**收获**：数据预处理是算法效果的前置条件，格式不统一的原始数据会让任何算法都失效。

### 10.2 算法选择：是否调大模型 API

**问题**：实验要求"自行实现算法"，但现代审核系统倾向调 LLM。

**解决方案**：
- 采用"双算法自研 + 大模型可选"架构
- 核心检索用 KeywordMatcher + TF-IDF 自实现（满足评分要求）
- 大模型仅在公文生成阶段可选调用（企业部署时）
- 知识库始终以 RAG 形态存在，不训练大模型（符合"大模型持续迭代"约束）

### 10.3 业务流程：强制路由 vs 自主提交

**问题**：初版设计考虑"按文档类型强制路由到法务"（合同必转法务）。

**解决方案**：改为业务自主提交。理由：
- 业务日常咨询未必到需法务阶段
- 强制路由增加法务负担
- 通过免责声明（"未提交法务不得作为决策依据"）规避合规风险

### 10.4 知识库数据结构演进

**问题**：初版设计为"结构化字段抽取"（合同主体/金额/期限），但业务更需"法律问题+操作指引"形态。

**解决方案**：
- 改为 FAQ 三段式（后扩为四段式含法条）
- 取消独立的结构化抽取模块，并入知识库
- 数据结构更轻、更易复用、更适合"出书"场景

### 10.5 案例链识别

**问题**：FAQ 中存在 Q64-67 这类连续追问案例，单条检索会割裂上下文。

**解决方案**：在 FAQ 条目中增加 case_id 字段，检索到一条时自动提示"本案例相关追问"。

### 10.6 企业留痕与归档合规

**问题**：初版 spec 仅考虑知识沉淀，未覆盖企业法务场景的审核留痕和电子档案归档合规要求。

**解决方案**：
- MVP 阶段在 WorkOrder 中预留完整留痕字段
- FAQ 条目增加 source_work_order_id，建立追溯链
- 完整 AuditLog 和 ArchivePackager 列入下一步计划（9.7、9.8 节）

### 10.7 从 CLI 到前后端分离的架构演进

**问题**：初版 spec 为 CLI 交互（input/print），无法满足企业实际使用需求。

**解决方案**：
- 升级为 Flask + 原生 HTML/JS 前后端分离架构
- 后端全 Python 满足实验 .py 要求
- 每个功能模块独立路由 + 独立前端页面
- 业务/法务/管理员三套差异化界面

### 10.8 三角色权限边界设计

**问题**：法务和管理员对知识库的操作权限需明确区分。

**解决方案**：
- 法务：可检索调用知识库，但只能通过"归档前审核+校验"流程入库，不能直接增删改
- 管理员：可直接 CRUD 知识库，拥有最高权限
- 新增 FaqValidator 校验器，法务入库前必须通过校验（硬错误阻止/软警告放行）
- 通过 @require_role 装饰器在路由层强制权限

### 10.9 脱敏与参考资产处理

**问题**：spec 中含特定企业字样，需泛化为"国有企业"通用方案。

**解决方案**：
- spec 正文统一使用"国有企业"表述
- jingtou-legal 作为可选参考资产保留（其公文模板含特定体系术语，企业部署时替换为目标企业制度体系）
- 核心算法和知识库结构与企业无关，可直接复用
